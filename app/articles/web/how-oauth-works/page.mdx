OAuth only made sense for me once I watched it solve a problem I actually had. This post is that problem:
voz.dev lets people sign up and save a short “message to future you,” and we want an outside site to read
that message _only_ when the user says so. OAuth is the bridge.

We’ll keep everything in TypeScript-friendly Next.js land. voz.dev (this site) becomes the OAuth provider
and resource server. The third-party app lives in `~/sandbox/oauth-third-party` as a vanilla Next App.
We’ll wire up the Authorization Code flow with PKCE, scopes, refresh tokens, and the consent screen copy
that actually tells the user what’s going on. All implementation details below assume the code will be
filled in later—the focus here is the architecture, routes, and data you’ll need.

## What we’re building

Picture three actors:

- **voz.dev** — both the authorization server (sign-in, consent, tokens) and the resource server (stores the message).
- **oauth-third-party** — a separate Next.js client that wants to display the message.
- **The user** — they own their words and must opt into sharing them.

The happy path looks like this:

1. The user signs up on voz.dev and leaves their message.
2. They visit oauth-third-party and click “Connect voz.dev.”
3. We redirect the browser to voz.dev’s `/oauth/authorize` endpoint with PKCE + state in tow.
4. The user signs in (if needed), reviews the consent screen, and approves the scope.
5. voz.dev issues an authorization code, which oauth-third-party swaps for tokens at `/oauth/token`.
6. oauth-third-party calls `/api/user-message` with the access token and shows the message.
7. The access token expires; oauth-third-party uses the refresh token to grab a new one.

We’ll lay down each part in order. If you want to see it all at once, skip to the “Putting it together” section.

## Step 0: Core data model on voz.dev

voz.dev already has a user table. Add two more tables (or Prisma models):

```sql
-- oauth_clients
client_id TEXT PRIMARY KEY,
client_name TEXT NOT NULL,
client_secret_hash TEXT NULL, -- null means public client, PKCE only
redirect_uris TEXT[] NOT NULL,
scopes TEXT[] NOT NULL,
created_at TIMESTAMP NOT NULL DEFAULT NOW()

-- oauth_authorizations
id UUID PRIMARY KEY,
user_id UUID NOT NULL REFERENCES users(id),
client_id TEXT NOT NULL REFERENCES oauth_clients(client_id),
scope TEXT[] NOT NULL,
code TEXT UNIQUE NOT NULL,
code_challenge TEXT NOT NULL,
code_challenge_method TEXT NOT NULL CHECK (code_challenge_method IN ('S256')),
expires_at TIMESTAMP NOT NULL,
consumed_at TIMESTAMP NULL
```

Store access and refresh tokens in a separate table (or Redis) if you want full revocation later. For this post,
assume the token table maps `token` → `{ user_id, client_id, scope, expires_at, type }`.

## Step 1: Capture the message during signup

The landing flow on voz.dev now collects a short blurb:

```tsx
// app/api/register/route.ts
export async function POST(request: NextRequest) {
  const { email, password, message } = await request.json()

  // 1. Hash the password, create the user, store the message.
  const user = await db.user.create({
    data: {
      email,
      passwordHash: await hashPassword(password),
      message: { create: { body: message } },
    },
  })

  // 2. Sign the session cookie and respond.
  return NextResponse.json({ ok: true }, { status: 201 })
}
```

Saving the message up front gives the OAuth provider something meaningful to protect. The resource server will
read from the `messages` table later.

## Step 2: Register an OAuth client for oauth-third-party

Seed a client in the database via a script or admin UI:

```ts
// scripts/register-oauth-client.ts
await prisma.oauthClient.create({
  data: {
    clientId: 'sandbox-third-party',
    clientName: 'Sandbox Message Viewer',
    clientSecretHash: hashSecret(process.env.NEW_CLIENT_SECRET!),
    redirectUris: ['http://localhost:4000/api/auth/callback'],
    scopes: ['message.read'],
  },
})
```

Drop the raw secret into your `.env.local` for oauth-third-party:

```
VOZ_CLIENT_ID=sandbox-third-party
VOZ_CLIENT_SECRET=super-long-random-string
VOZ_OAUTH_BASE_URL=http://localhost:3000
```

By treating the Next app as a confidential client (server-side exchange), we can accept both PKCE and client secrets.

## Step 3: `/oauth/authorize` — redirect + consent

Time to mint authorization codes. This endpoint needs to:

1. Validate the incoming query (`client_id`, `redirect_uri`, `scope`, `response_type=code`, `code_challenge`, `state`).
2. Prompt the user to sign in if they’re not already.
3. Show a consent screen summarizing the scope (e.g. “Read your saved voz.dev message”).
4. On approval, persist an authorization record and redirect back with `?code=...&state=...`.

```tsx
// app/oauth/authorize/route.ts
export async function GET(request: NextRequest) {
  const params = request.nextUrl.searchParams
  const client = await findClient(params.get('client_id'))

  assertRedirectUri(client, params.get('redirect_uri'))
  assertScope(client, params.get('scope'))
  requirePKCE(params)

  const session = await requireLogin(request)
  if (!session.consentedTo(client.id, params.get('scope'))) {
    return renderConsentScreen({ client, scope: params.get('scope') })
  }

  const authorization = await createAuthorization({
    userId: session.userId,
    clientId: client.id,
    scope: params.get('scope')!.split(' '),
    codeChallenge: params.get('code_challenge')!,
    method: params.get('code_challenge_method')!,
  })

  const redirectUri = buildRedirectUri({
    base: params.get('redirect_uri')!,
    code: authorization.code,
    state: params.get('state'),
  })

  return NextResponse.redirect(redirectUri)
}
```

Persist `code_challenge` and `code_challenge_method` so you can verify PKCE during the token exchange.

## Step 4: `/oauth/token` — exchange code for tokens

This handler is the workhorse. It must:

- Authenticate the client (basic auth header or form fields).
- Validate the authorization code, PKCE verifier, and expiration window.
- Issue an access token (short expiry) and a refresh token (longer expiry).
- Invalidate the authorization code so it can’t be reused.

```ts
// app/oauth/token/route.ts
export async function POST(request: NextRequest) {
  const body = await request.formData()
  const { client, authCode } = await authenticateTokenRequest(body)

  verifyPKCE({
    codeChallenge: authCode.codeChallenge,
    verifier: body.get('code_verifier') as string,
  })

  const tokens = await issueTokens({
    userId: authCode.userId,
    clientId: client.id,
    scope: authCode.scope,
  })

  await markAuthorizationAsUsed(authCode.id)

  return NextResponse.json({
    token_type: 'Bearer',
    access_token: tokens.accessToken.value,
    expires_in: tokens.accessToken.expiresInSeconds,
    refresh_token: tokens.refreshToken.value,
    scope: tokens.scope.join(' '),
  })
}
```

Access tokens can be JWTs signed with `HS256` or opaque strings stored server-side. Either way, include the
user ID, client ID, scope, and expiry.

## Step 5: `/api/user-message` — the protected resource

The resource server logic runs like any authenticated API route: check the Bearer token, make sure it contains
`message.read`, and fetch the message.

```ts
// app/api/user-message/route.ts
export async function GET(request: NextRequest) {
  const token = extractBearerToken(request.headers.get('authorization'))
  const payload = await verifyAccessToken(token)

  assertScope(payload, 'message.read')

  const message = await db.message.findUnique({
    where: { userId: payload.userId },
  })

  if (!message) {
    return NextResponse.json({ error: 'not_found' }, { status: 404 })
  }

  return NextResponse.json({
    body: message.body,
    updatedAt: message.updatedAt.toISOString(),
  })
}
```

Token failures return `401` with a `WWW-Authenticate` header so the client knows to refresh.

## Step 7: Kick off the authorize redirect

The home page offers a button that builds the OAuth URL. Generate PKCE + state in the browser, stash them
in `localStorage` (or encrypted cookies), and redirect.

```tsx
'use client'

export function ConnectButton() {
  const handleClick = async () => {
    const { verifier, challenge } = await createPkcePair()
    const state = crypto.randomUUID()

    sessionStorage.setItem('pkce_verifier', verifier)
    sessionStorage.setItem('oauth_state', state)

    const authorizeUrl = new URL('/oauth/authorize', process.env.NEXT_PUBLIC_VOZ_OAUTH_BASE_URL)
    authorizeUrl.searchParams.set('client_id', process.env.NEXT_PUBLIC_VOZ_CLIENT_ID!)
    authorizeUrl.searchParams.set('redirect_uri', `${window.location.origin}/api/auth/callback`)
    authorizeUrl.searchParams.set('response_type', 'code')
    authorizeUrl.searchParams.set('scope', 'message.read')
    authorizeUrl.searchParams.set('state', state)
    authorizeUrl.searchParams.set('code_challenge', challenge)
    authorizeUrl.searchParams.set('code_challenge_method', 'S256')

    window.location.href = authorizeUrl.toString()
  }

  return <button onClick={handleClick}>Connect voz.dev</button>
}
```

Expose `NEXT_PUBLIC_VOZ_*` variants of the env variables for client-side consumption.

## Step 8: Handle `/api/auth/callback` in oauth-third-party

Once the user lands back with `?code=...`, trade it for tokens on the server.

```ts
// app/api/auth/callback/route.ts
export async function GET(request: NextRequest) {
  const params = request.nextUrl.searchParams
  assertState(params.get('state'))

  const tokenResponse = await fetch(`${process.env.VOZ_OAUTH_BASE_URL}/oauth/token`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/x-www-form-urlencoded',
      Authorization: basicAuth(process.env.VOZ_CLIENT_ID!, process.env.VOZ_CLIENT_SECRET!),
    },
    body: new URLSearchParams({
      grant_type: 'authorization_code',
      code: params.get('code')!,
      redirect_uri: `${request.nextUrl.origin}/api/auth/callback`,
      code_verifier: readVerifierFromSession(request),
    }),
  })

  const tokens = await tokenResponse.json()
  await persistSessionCookie({
    accessToken: tokens.access_token,
    refreshToken: tokens.refresh_token,
    expiresAt: Date.now() + tokens.expires_in * 1000,
  })

  return NextResponse.redirect(new URL('/dashboard', request.url))
}
```

Persist the verifier somewhere the server can read (signed cookie, encrypted storage). After success, clean it up.

## Step 9: Display the message

The dashboard fetches from an internal API route that proxies the access token to voz.dev. Refresh when needed.

```ts
// app/api/message/route.ts
export async function GET() {
  const session = await readSessionCookie()
  const response = await fetch(`${process.env.VOZ_OAUTH_BASE_URL}/api/user-message`, {
    headers: { Authorization: `Bearer ${session.accessToken}` },
  })

  if (response.status === 401) {
    const refreshed = await refreshSession(session.refreshToken)
    return NextResponse.json(refreshed.message)
  }

  return NextResponse.json(await response.json())
}
```

In `/dashboard/page.tsx`, call that handler via `fetch('/api/message', { cache: 'no-store' })` inside `getMessage()`
and render optimistic UI states. Remember to show helpful copy if the user hasn’t written a message yet.

## Hardening for production

- **Token storage** — Store refresh tokens hashed, and rotate them on every use. Invalidate the whole chain if one leaks.
- **Revocation endpoint** — Support `POST /oauth/revoke` so users can disconnect clients.
- **Scopes & defaults** — Default to the least powerful scope. Expand later for write access or profile reads.
- **Logging** — Log grants, denials, and token exchanges for audit trails. Mask personally identifiable info.
- **Consent UX** — Surface which redirect domain is asking for access, and how often it can refresh tokens.
- **Monitoring** — Add alarms for suspicious refresh activity (e.g. repeated failures, access from new geo/IP pairs).

## Putting it together

By the end of this walkthrough you have:

- voz.dev acting as a first-party OAuth provider with authorization, token, and resource endpoints.
- A third-party Next.js client that uses PKCE + refresh tokens to read a user’s message.
- Database scaffolding to track clients, codes, tokens, and user consent.

Implementation is “just” wiring up the handlers, but the blueprint above keeps you from forgetting the
important guardrails. When the time comes to ship, grab screenshots of the consent screen and run through the
